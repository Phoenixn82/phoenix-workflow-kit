A field guide to a working AI workflow

The Workshop.

One operator, a router of AIs, and a rack of right-sized tools. The principle: send every job to the role that fits it — think, build, read, bulk — and let a vault remember everything. The author fills those roles with Claude, Codex, Gemini & a local model router; you map the same roles onto whatever you run. A built-in prompt teaches your AI to take only the parts that fit you.

21 mechanics ~120 wrenches 4 routed lanes · any AI 1 canonical vault portable · secret-free

The idea

Stop reaching for the same hammer.

Most AI setups are a junk drawer: one giant prompt, a pile of half-remembered tricks, and a model that improvises every time. This one is a workshop. Every job gets the right-sized tool, chosen on purpose — and the tools know about each other.

01

Right-sized surfaces

A narrow ask grabs a single wrench. A coordinated job calls a mechanic that dispatches its wrenches. A whole project from one prompt calls the apex. You never spin up more than the task needs.

02

Roles, not brands

Route each turn to the right lane — thinking, building, long-context reading, cheap bulk — not to a favorite vendor. The author fills those lanes with Claude, Codex, Gemini & a local router; you map them to whatever AIs you run.

03

Nothing runs on its own

No timers, no cron, no background spend while you're away. Every action is triggered by you. Autonomy is real — but it's summoned, never ambient.

04

The vault is the memory

A second-brain vault is the single source of truth: project status, decisions, open loops, standing rules. Sessions are disposable; the vault is forever.

05

Cost-first, always

Cheapest capable tool wins. Scraping climbs a ladder — managed API, then stealth browser, then local drivers — only escalating when it must. Free local models soak up bulk work.

06

Built to work every time

A reliability standard governs anything shipped: it must resume, verify, and survive a cold restart. "It worked once" is not done. Done is proven.

The anatomy

Apex, mechanic, wrench.

Three altitudes. You pick the one that matches the job — and a router helps when the choice isn't obvious.

Apexproject-orchestrator

A whole project from a single prompt. Intake → scope → dispatch → completion, coordinating every mechanic below it. The thing you call when you say "just build me the app."

Mechanicscoordinators

Each owns a domain and dispatches its own wrenches: build, ship, seo, design-studio, plan-room, web-scrape, second-brain, router, guard, and more. Call a mechanic when a job needs several tools in concert.

Wrenchessingle tools

The ~120 leaf tools that do one thing well — scrape, land-and-deploy, autoplan, schema, recall. Loaded only when actually used, so the context stays lean. Call one directly for a narrow ask.

The router

Lanes, not loyalties.

The core habit isn't "use Claude" or "use Codex" — it's routing each turn to the role that fits it. The author fills these four lanes with the AIs they happen to run; the pattern is vendor-agnostic. You map the same roles onto whatever you use.

Lane 1 · think

The architect

Planning, system design, decisions, prompt & skill orchestration, reviewing whether work matches intent.

Author runs: Claude  ·  You'd map: your strongest reasoning model

Lane 2 · build

The builder

Writing code & tests, mechanical refactors, running commands, code-quality review, long hands-free /goal loops.

Author runs: Codex  ·  You'd map: your best coding agent

Lane 3 · read

The long-context reader

Huge documents, whole codebases, video/audio — one giant context window. Reads & analyzes; doesn't write the code.

Author runs: Gemini (1M ctx, multimodal)  ·  You'd map: your widest-context model

Lane 4 · bulk & cheap

The free local router

High-volume, low-stakes work that shouldn't burn premium tokens — routed to free/local models, OpenAI-compatible, strongest-healthy-first.

Author runs: FreeLLMAPI (local :3001)  ·  You'd map: a local/free model, or skip it

One AI for everything? It simply wears all four hats — the routing logic still tells it which mode to be in. Codex-primary? The think + build lanes both point at Codex. Running something niche that can only chat and can't touch files? Keep the ideas — the routing, the prompts, the vault discipline — and skip the machinery it can't run. The roles are fixed; who fills them is yours.

The toolbox

Twenty-one mechanics.

Grouped by how they load. You don't memorize these — the index and decision-map route you. Here's the whole rack at a glance.

Always-core — oriented at every session start

router

Routes work to the right brain/lane — Claude, Codex, Gemini, or the local free-model router — with a no-self-rescue rule that hands off instead of looping.

second-brain

The vault: capture, recall, learn, end-session. Persistent memory of projects, decisions, and standing rules across every session.

guard

Opt-in safety: careful warns before destructive commands, freeze locks edits to a scope. Never nags unless you turn it on.

Standalone keepers — called directly

cso

Chief security officer. Read-only deep security review across a dozen phases — never edits, just finds.

investigate

Systematic debugging: root cause before any fix. No symptom whack-a-mole.

context-audit

Audits what's eating the context window and what to trim — quality first, noise out.

windows-launcher

Ships clickable desktop launchers for apps — a branded icon, not a "run pnpm dev" note.

printing-press-router

Enforces the CLI > API > MCP ladder before touching any third-party service.

On-demand mechanics — loaded only when the task calls

build

Scaffolds sites, apps, and backends with the right runtime per signal — never a webview shell by reflex.

mobile-app

The Expo / React-Native pipeline: scaffold, on-device test, Supabase wiring, ship to stores.

ship

Land-and-deploy, canary, benchmark, QA, release docs. The path from "works locally" to "live."

seo

Full-site audit that fans out specialist sub-agents — technical, content, schema, GEO, backlinks, local.

design-studio

DRIP pipeline: brand DNA × layout archetype × stack, rotated so no two sites look the same.

plan-room

Autoplan, decision toolkit, and CEO / eng / devex review lenses before a line is written.

web-scrape

Cost-first scraping: Firecrawl → Cloak stealth browser → local drivers, escalating only on block.

content-forge

Marketing copy, humanizer, fact-checker, prompt-master — content that doesn't read like a robot.

morning-briefing

On-demand synthesis from the vault + feeds. Never scheduled; fired when you want the picture.

video-scan

Pulls transcript, metadata, and signal out of a video so you don't have to watch it.

skill-forge

The meta-mechanic: scout for a matching skill, draft new ones, mirror them to Codex.

The bridge

Two AIs that know what the other did.

The hard part of running more than one AI is keeping them in sync. A low-token bridge makes the connection bidirectional — neither works blind. (The author's bridge links Claude & Codex; the same shared-log + hand-off pattern works between any two agents you run — or collapses to nothing if you use a single AI.)

Shared activity log

One append-only line per action in .claude-codex-log.md. A hook writes Claude's edits automatically; a turn-end notify writes Codex's. Each side tails it and reconciles before touching a file the other just changed.

Hand-off requests & manifests

Either AI can hand a task to the other with a structured request; session results land as relayable manifests so nothing gets lost between contexts.

Tool parity

Before either side is told to use a tool, a parity checker confirms the other can actually reach it — so a dispatch never assumes a capability that isn't there.

Autonomous goals, hands-free

The dispatcher hands long jobs to Codex's /goal loop via headless codex exec — it runs to completion with no keystroke, governed by a budget and a verification gate. A cross-session lock keeps two AIs from clobbering each other's files.

Make it yours

The implementation prompt.

Here's the point of the whole kit. Download the framework zip, then paste this prompt into whatever AI you use along with it. It won't blind-copy anyone's setup — it asks what AIs you run (Claude, Codex, Gemini, GPT, a local model, or something obscure), maps the framework's roles onto your lineup, rewrites the files to match your primary AI, takes only what your AI can actually use, and finishes with a best-practices cheat-sheet for operating it.

paste into your AI (any) — attach phoenix-workflow-framework.zip 
I'm adopting another power-user's AI workflow system. Attached is phoenix-workflow-framework.zip — a "mechanic / wrench" skill framework with a role-based router, a cross-AI bridge, hooks, and a lock layer. It is a TEMPLATE, not a drop-in. It's written around the AUTHOR's specific AI lineup (a thinking lane = Claude, a building lane = Codex, a long-context reading lane = Gemini, a cheap/bulk lane = a local model router), and the paths are their machine. The private vault and all secrets were left out.

The philosophy is vendor-agnostic — the ROLE matters, not the brand. Your job is to adapt it to MY AIs, whatever they are, and implement only what's useful to me. Do NOT copy anything yet. First, interview me.

STEP 1 — Learn my actual setup. Ask a few at a time, wait for answers, and ask follow-ups until you genuinely understand my lineup. Do NOT assume I use Claude or Codex.
- List every AI/LLM I use — Claude, Codex, GPT, Gemini, local/open models, or anything niche or obscure (even something like "owl alpha"). For EACH: what is it good at, and what can it actually DO — write files? run shell commands? use tools/MCPs? or only chat? Which one is my daily driver?
- Which of my AIs (if any) would fill each role: deciding/architecting · writing code · reading long docs or video · cheap bulk work. One AI can wear several hats — that's fine; say so.
- My OS and shell (paths + hooks differ across Windows / macOS / Linux).
- The kinds of projects I work on, and what I repeat often enough to want templated.
- What's broken or annoying in my current workflow.
- Whether I want a persistent "second brain" vault, and where it'd live.
- My autonomy comfort, from approve-every-step to fully hands-free.
- What tools / MCPs / CLIs I already have, and which providers I pay for.
- Solo, or do multiple AIs/people touch the same repos?

STEP 2 — Map the framework onto MY AIs. Read AGENTS.md, SKILLS_INDEX.md, DECISION_MAP.md and MANIFEST.md to learn the philosophy and the role-based router. Then translate the author's role assignments onto MY lineup and show me a ranked proposal:
- which mechanics / wrenches / hooks / patterns help MY work, and which to skip;
- how each role maps to one of MY AIs — e.g. "you're Codex-primary, so think + build both route to Codex"; or "your main model can't run hooks, so keep the skill docs + vault discipline and drop the PowerShell machinery";
- for any AI that can only chat (no file/command access), keep the IDEAS — the routing logic, the prompts, the vault habit — and drop the machinery it can't execute.
Then wait for my approval.

STEP 3 — Implement the approved subset, rewritten for me. Rewrite the files so they match MY primary AI and machine: swap the author's lane assignments for mine throughout the docs/skills, rewrite every hardcoded path for my OS/username, and translate or drop hooks/scripts my AI can't run. If I'm Codex-primary, rewrite the Claude-lane wording to Codex (and vice-versa); if I'm on an obscure model, grab only what it can genuinely use. Never introduce secrets. Work in small phases; after each, say exactly what changed and how to check it.

STEP 4 — Verify. Show me each thing working with a concrete example from MY actual work, plus a one-line "how do I know it's working" check.

STEP 5 — Best-practices handoff. Once it's built on my machine, give me a short "how to operate your new workflow" cheat-sheet so I get the best results:
- right-size the call — a wrench for a narrow ask, a mechanic for a coordinated job, the apex for a whole project; don't over-summon;
- route by role every turn — pick the lane/AI that fits the task;
- keep the vault current — capture decisions + status as you go, or the memory rots;
- trigger, don't automate — nothing runs on its own; you summon it;
- verify before "done" — show evidence, never "it should work";
- cost-first — prefer the cheapest capable lane; push bulk to the free/local one;
- start small — adopt 2–3 pieces, live with them a week, then add more.

Rules: surgical changes, explain trade-offs, ask before anything irreversible, adapt to me rather than cloning the author. Begin with STEP 1 now.
