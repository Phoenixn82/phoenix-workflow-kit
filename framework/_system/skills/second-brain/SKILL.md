# second-brain — memory engine

Operates the user's vault as a self-rewriting AI-first knowledge base. Inspired by Karpathy's LLM Wiki pattern: new information rewrites existing pages, contradictions reconcile automatically, and the vault is built for **future-Claude retrieval** — not human readability.

**Always-core** (loads every session). But scoped — only the index + active-project section load at start; the rest is on-demand.

---

## Where the vault lives

- **Canonical vault (this rebuild forward):** `Desktop/AI_Projects/_system/second-brain/`
- **Harness-loaded layer** (auto-loaded by Claude Code at session start): `~/.claude/projects/<project-slug>/memory/`
- **Archived old vault:** `Desktop/AI_Projects/_archive/claude_vault_2026-05-pre-rebuild/` (the old 101 MB `claude_projects/claude_vault/` — moved here in Phase 7, never auto-touched)

The mechanic operates the canonical vault. The harness layer is a per-session cache — second-brain mirrors active items into it so Claude Code auto-loads them at start without reading the whole vault.

## The three axes

### Project axis: `_system/second-brain/Projects/<slug>/`

One folder per active project. Each folder contains:

```
Projects/<slug>/
  _README.md          # What this project is, current status, north star
  decisions.md        # Dated decisions with reasoning ("why X over Y")
  status.md           # Live snapshot — what's done, in flight, blocked
  errors.md           # what / cause / fix / prevention entries
  tokens.md           # expensive-approach → cheaper-path entries
  open-loops.md       # Things waiting on something (the user, third party, etc.)
  next-prompt.md      # Generated by end-session ritual; exact prompt for next session
```

Active projects = whatever folders currently live under `Projects/` (enumerate that directory or read `index.md` for the live list — do NOT hardcode, the portfolio changes). These are the protected projects — second-brain reads/writes ABOUT them but never edits their source code; that's `guard`'s job.

### Action axis: `_system/second-brain/Actions/`

Cross-project preferences and defaults — the "do it the way I asked last time" memory. One file per preference cluster.

```
Actions/
  scraping-tier.md         # Firecrawl → Cloak Browser → local drivers (cost-first)
  design-defaults.md       # "always grab the photos too" etc.
  brevity-defaults.md      # "no em dashes, no filler, terse recaps"
  code-discipline.md       # Karpathy guidelines, surgical changes, blast radius
  routing-defaults.md      # Codex is default code lane; Gemini doesn't write code files
  error-handling.md        # No fixes without root cause (Iron Law from `investigate`)
  ...
```

When the user tells me to do something a certain way more than once, it becomes a stored action-default — no re-asking, same session or future.

### Mechanic-state axis: `_system/second-brain/Mechanics/<mechanic>/state.md`

The "always-current map of what's in the workflow" — one file per mechanic that future-Claude reads BEFORE deciding to add a new skill, propose a new mechanic, or refactor existing ones. Per AGENTS.md hard rule #11 (future-Claude actively maintains the skill stack), this is where the current state of every mechanic lives.

```
Mechanics/
  router/state.md          # current wrenches, recent additions, known gaps, trigger patterns
  second-brain/state.md
  guard/state.md
  build/state.md
  web-scrape/state.md
  ship/state.md
  ...
  project-orchestrator/state.md
  video-curator/           # exception: carries _README.md + learnings.md instead of a single state.md
```

Each `state.md` carries:
- **Mechanic name + purpose** (one paragraph)
- **Current wrenches** (one line each, with brief "what it does")
- **Recent additions / changes** (dated entries, last ~10)
- **Known gaps the user has surfaced** (things this mechanic doesn't cover that came up but weren't added yet)
- **Trigger phrase patterns** this mechanic owns (so future-Claude knows when to route here)
- **Cross-mechanic dependencies** (which other mechanics this one calls, which call it)
- **Open considerations** (decisions deferred, ambiguous cases, things to think about)

**Updated by:**
- `skill-forge` mechanic when it scaffolds a new skill or refactors an existing one (auto-writes the addition)
- The end-of-session ritual (category 6: new skills found + where they integrate writes back here)
- `skill-scout` when it surfaces a recurring trigger pattern with no good owner
- The user manually when needed

**Read by:**
- Future-Claude during the skill-intake flow (Step 0: "read the live mechanic state before deciding the shape")
- `skill-scout` when matching a query against existing capabilities
- `context-audit` when checking for orphan skills or overlapping trigger patterns
- The end-session ritual when deciding which mechanic's state needs updating

The point: never propose a "new" skill that overlaps an existing wrench because future-Claude didn't know it existed. The mechanic state files are the source of truth for "what's already there".

## Dev registry: `_system/second-brain/dev-registry/`

Cross-project local-dev coordination. The original system had a great pattern: when the user runs multiple projects locally at the same time (Example Co + Example Gallery + Example App, say), localhost ports don't collide because each project has a stable assigned port from a central registry.

This carries forward. The registry lives in second-brain so it's persistent project state.

```
dev-registry/
  ports.md           # the canonical port table — one row per project (the only seeded file today)
  databases.md       # local DB ports + connection strings per project — created on demand (not yet seeded)
  services.md        # any other shared local services (Redis, queue workers, etc.) — created on demand (not yet seeded)
```

### `ports.md` format

The live file at `dev-registry/ports.md` is the format authority — read it before writing rows. It is a **5-column** table (`Port | Project slug | Service | Source | Notes`), where `Source` is `baked` (port hardcoded in config/.env/compose) vs `launcher-injected` (framework default, launcher passes `-p`):

```markdown
---
type: dev-registry
date-updated: 2026-05-29
ai-first: true
---

# Local dev port assignments

| Port | Project slug | Service | Source | Notes |
|---|---|---|---|---|
| 3000 | next.js-default | (reserved) | — | leave free; Next.js apps need `-p` injection |
| 3001 | freellmapi | FreeLLMAPI router | always-on | the user's local LLM proxy |
| 3010 | example_client | Frontend (Next.js) | launcher-injected | `next dev -p 3010`; no API yet |
| 5180 | example_app | Frontend (SvelteKit) | **baked** | `vite.config.ts` + docker-compose |
| 8000 | example_app | API (FastAPI) | **baked** | docker-compose `8000:8000` |
| 8040 | example_scraper/example-scraper | API (FastAPI) | baked* | remapped from 8000 to end the example_app collision |

- **Next free frontend (3000s):** 3090
- **Next free API (3000s):** 3091
```

### Assignment flow

When the `build` mechanic scaffolds a new project that runs locally:

1. **Read** `_system/second-brain/dev-registry/ports.md`
2. **Find** the next free port (per the "Next free port" hint at the bottom; verify by table scan)
3. **Assign** the port to the new project; if the project has both frontend + API, assign two ports paired
4. **Write back** to `ports.md`: add the row(s), update the "Next free port" hint
5. **Bake** the port into the project's config (Vite/Next/SvelteKit config + `.env.local` + any docker-compose). The project must always start on the same port — the user shouldn't have to remember which port goes to which project.

### When a project runs

The dev launcher (Windows-specific patterns in `windows-launcher` standalone keeper) reads the registry to know which port to open in the browser. The user's "open project X" command should land on the right localhost URL without him remembering.

### Conflicts

If the assigned port is already in use when launching:
- The launcher pings the port first
- If something's responding, surface to the user: *"port 3010 is busy — is example_client already running, or is another process on that port?"*
- Don't auto-reassign without asking — port stability is a feature, not a fallback

### Multi-machine concerns

The registry is the single source of truth at `_system/second-brain/dev-registry/ports.md`. (OneDrive sync is gone as of 2026-05-28; to stay consistent across machines, sync the `AI_Projects` folder via git or another mechanism, not OneDrive.)

On-demand reference notes that aren't project-scoped, action-defaults, or mechanic-state — e.g. `reliability-standard.md` (the load-bearing standard wired into AGENTS.md hard rule #12), and course/transcript notes. Loaded only when explicitly relevant, not at session start.

## AI-first rule (non-negotiable)

Every note written to the vault MUST follow this. The vault is for **future-Claude**, not for human review.

1. **Self-contained context.** Each note explains itself. Don't rely on backlinks alone.
2. **"For future Claude" preamble** — 2–3 sentence summary at the top so Claude can decide relevance in 10 seconds.
3. **Rich frontmatter.**
   ```yaml
   ---
   type: decision | error | preference | status | error-log | token-note | person | idea | next-prompt
   project: <slug>           # for project-axis notes
   date: 2026-05-27
   tags: [tag1, tag2]
   ai-first: true
   confidence: stated | high | medium | speculation
   ---
   ```
4. **Recency markers per claim.** "Codex Pro $200/mo (as of 2026-05-27, OpenAI)" so future-Claude knows what to verify before relying on it.
5. **Sources preserved verbatim.** Every external claim has its source URL inline. Every quote keeps the verbatim form.
6. **Cross-links mandatory.** Every person / project / idea / decision uses `[[wikilinks]]`. Link liberally — a `[[name]]` that doesn't match an existing note yet is fine; it marks something worth writing later.
7. **No "humanizing".** Don't strip frontmatter or preambles to make notes look cleaner. Vault notes are not blog posts.

This rule lives at `_system/second-brain/_CLAUDE.md` (the vault's operating manual). It's enforced on every wrench's write path.

## Scoped auto-pull

Second-brain is always-core, but it does NOT load the whole vault at session start. The load is bounded:

**Always loads at session start (~1-3 KB total):**
- `_system/second-brain/_CLAUDE.md` (vault operating manual)
- `_system/second-brain/index.md` (catalog — one line per top-level node)
- ALL of `Actions/*.md` (preferences are small, all relevant cross-project)

**Loads on project mention:**
- When the user names a project second-brain hasn't loaded this session, read `Projects/<slug>/_README.md` + `status.md` + most-recent `next-prompt.md` BEFORE responding.

**Loads on explicit request:**
- "What do I know about X" → `recall` wrench searches the vault
- "Show me the errors on Y" → loads `Projects/<slug>/errors.md`
- "Remind me what I decided about Z" → loads `Projects/<slug>/decisions.md`

The bound is intentional. Loading 101 MB of vault at session start is exactly the noise-bloat the rebuild is fixing.

## Vault access methods

**Method A (preferred): MCP server `mcp-obsidian`.**
If MCP tools (`get_file_contents`, `list_files_in_vault`, `search`, `append_content`, `write_file`) are available, use them. Faster on large vaults.

```bash
claude mcp add obsidian-vault -s user -- npx -y mcp-obsidian "C:/Users/<you>/Desktop/AI_Projects/_system/second-brain"
```

**Method B (fallback, always works): direct filesystem.**
Standard Read / Write / Edit / Glob against the vault path. The vault is plain markdown — every operation works without MCP, just more verbosely.

Use whichever is available; never bother the user about which one.

## End-of-session ritual (the user triggers)

When the user says "end session", "save my work", "wrap up", "extract takeaways", or similar, run the full ritual. See [wrenches/end-session.md](wrenches/end-session.md) for the step-by-step. Summary of what gets extracted:

1. **Decisions** — every choice the user made and the reasoning
2. **Preferences / defaults** — anything that should now be a standing rule (writes to `Actions/`)
3. **Errors with root-cause + fix + prevention** — writes to `Projects/<slug>/errors.md`
4. **Token-expensive approaches** — what burned tokens and the cheaper path next time → `Projects/<slug>/tokens.md`
5. **Project status changes** — what advanced, what's now blocked → `Projects/<slug>/status.md`
6. **New skills found and where they integrate** — per the Skill Law, every new skill gets a home
7. **Open loops** — things waiting on something or someone → `Projects/<slug>/open-loops.md`
8. **Corrections** — every time the user corrected my behaviour, what the correction was and why
9. **Generates `next-prompt.md`** — the exact prompt to start the next session with so the work transfers perfectly

The mechanic shows a DRAFT first. The user approves or trims. Only approved content writes to disk.

**What never gets written:**
- Raw code
- Trivia or junk nodes
- An error without its fix
- Anything not human-readable + AI-readable

If a note doesn't earn both, it doesn't get a node.

## Error memory: what / cause / fix / prevention

Every error hit during work becomes an `errors.md` entry in the relevant project folder. Format:

```markdown
## 2026-05-27 — Snapshot tar exit code 255 despite success

**What:** PowerShell tar command for snapshot returned exit code 255 even though the archive completed successfully (5.85 GB, manifest verified).

**Cause:** PowerShell `Pop-Location` after the `try { } finally { }` block emitted a non-zero status that bubbled up. The tar binary itself returned 0.

**Fix:** Check the log file's "Tar exit code:" line and archive file size, not the PowerShell wrapper exit code, to determine actual snapshot success.

**Prevention:** When wrapping CLI calls in PowerShell with cleanup logic, always log the inner command's exit code separately. Don't trust the wrapper's exit code as the source of truth.

**Confidence:** high (observed in this session)
**Links:** [[example-project]] [[snapshot-2026-05-27]]
```

Same error pattern showing up twice across sessions → second-brain catches it and reminds Claude/the user that this was solved before.

## Token memory: expensive → cheaper path

When an approach burned more tokens than necessary, the cheaper path gets recorded in `Projects/<slug>/tokens.md`. Format:

```markdown
## 2026-05-27 — Reading whole files when partial would do

**Expensive:** Read entire 1434-line `codex/SKILL.md` to extract the 3-mode wrapper logic. Burned ~25K tokens of preamble bloat.

**Cheaper:** Read with `offset=850, limit=200` to skip the gstack preamble and get straight to the wrench-relevant content.

**When to apply:** Reading any gstack-family skill (~30 skills have the ~50-100 line bash preamble). Always skip past line ~200 unless the preamble itself is the target.

**Token saved per skill read:** ~20-30K
**Confidence:** high (observed this session)
```

Token self-audit is on-demand or fires automatically past ~500-600K session tokens (so the audit itself doesn't become the waste).

## Handoff files

After the end-session ritual, second-brain emits a handoff file at `Projects/<slug>/handoff-<date>.md`. The handoff is for the next session OR for a different destination (a new Claude session, a Codex worker, the user coming back tomorrow). Format:

```markdown
---
type: handoff
project: example-project
from-session: <session-id>
date: 2026-05-27
---

## For future Claude

<One-paragraph orientation: what this project is, what is done, what is in progress, and where the plan/spec docs live. Keep it tight — future-Claude reads this first.>

## State at end of session

- Phase 3 progress: router done, second-brain in progress, guard + 10 on-demand mechanics still ahead
- Snapshot at `_AI_Projects_Rebuild_Snapshot/claude_projects_snapshot_2026-05-27.tar` (5.85 GB)
- agentic-os hooks paused; folder preserved at `~/agentic-os/`
- 27 skills cut, 5 standalone keepers identified, ~99 fold into mechanics

## Exact next prompt

> "Pick up Phase 3 second-brain mechanic. Last file written: <path>. Next file: <path>. Plan is at [path]."

## Open loops

- [[guard mechanic]] not yet built
- 10 on-demand mechanics to build
- Codex helper scripts for router (track_attempt.py, log_decision.py, preflight.py) — flagged for Codex dispatch in Phase 5/6
```

The orchestrator (when built in Phase 6) triggers handoff emission automatically on scope completion. The user can also trigger manually.

## Harness memory integration

The current Claude Code harness loads memory from `~/.claude/projects/<project-slug>/memory/MEMORY.md` at session start. Second-brain mirrors active items into this layer so the harness picks them up automatically.

**Sync direction (vault → harness):** when a project becomes active, second-brain syncs key entries:
- Active feedback memories (from `Actions/*.md`) → harness `feedback_*.md` files
- Current project status (from `Projects/<slug>/status.md`) → harness `project_status.md`
- Active error patterns (from `Projects/<slug>/errors.md`) → harness `error_*.md` files

**Sync direction (harness → vault):** end-session ritual reads any new harness memories written during the session and writes them as canonical vault entries.

**Why both layers?** The harness layer is what Claude Code's `auto memory` system auto-loads at session start — small, indexed, current. The vault is canonical, durable, full. Two layers means session start is cheap (harness only) AND the long-tail history is preserved (vault).

---

## What second-brain does NOT do

- It does not auto-write during normal work. Mid-session captures happen via the `capture` wrench when the user asks. The end-session ritual is on the user's command.
- It does not load the whole vault. Scoped auto-pull means only the index + active project + actions load by default.
- It does not write raw code, trivia, or errors without fixes. If a note isn't human-readable AND AI-readable, it doesn't get a node.
- It does not maintain the OLD `claude_vault/` content. That's archived. New vault is canonical.
- It does not modify protected project source code (that's `guard`'s job). Second-brain only writes ABOUT projects, never their internals.

## Wiring (Phase 4 will do this, not Phase 3)

Phase 4 of the rebuild builds the actual vault skeleton + the sync scripts. Phase 3 (now) just lands the mechanic + wrench SKILL.md files describing how it works. When Phase 4 runs, it:

1. Scaffolds `_system/second-brain/_CLAUDE.md` (vault operating manual derived from this SKILL.md)
2. Scaffolds `_system/second-brain/index.md` (initial catalog)
3. Scaffolds `Projects/<slug>/` folders for each active protected project
4. Scaffolds `Actions/` with current preference defaults (scraping-tier, brevity, claude-vs-codex split, etc.)
5. Wires the harness sync (~/.claude/projects/<slug>/memory/ ↔ vault)
6. Optionally wires mcp-obsidian if the user wants the MCP path
7. Archives old `claude_vault/` to `_archive/`
